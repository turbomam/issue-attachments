## WAN SSH to NUC — Troubleshooting

Attempted to SSH into the Ubuntu NUC (192.168.0.204) from girlfriend's house using `ssh nuc-wan`. Connection hung with no response.

### SSH Config Entries (both use the same DDNS hostname)
- `nuc-wan` — basic WAN SSH to NUC
- `nuc-public` — WAN SSH with Samba tunnel (LocalForward 4450→445)

### Diagnostics

| Test | Result |
|------|--------|
| `nslookup mamillerpa.tplinkdns.com` | Resolves to 71.224.216.71 |
| `ping 71.224.216.71` | Responds, ~21ms RTT |
| `nc -zv -w 5 mamillerpa.tplinkdns.com 22` | Timed out |
| Probed ports 2222, 2200, 8022, 22222, 443, 8080 | All timed out |
| `curl ifconfig.me` (from girlfriend's network) | 73.81.142.250 (confirms we're on a different network) |

### Diagnosis

The IP `71.224.216.71` pings but has zero open ports — it's almost certainly **no longer your home's public IP**. The ISP rotated the IP and the TP-Link router's built-in DDNS client didn't update `mamillerpa.tplinkdns.com`. The hostname now points to someone else's device.

This was working ~1 week ago from an urgent care office. The NUC itself is confirmed running (used locally within the hour).

### TODO (when home)
1. Check router WAN IP at 192.168.0.1 — confirm mismatch with 71.224.216.71
2. Fix/restart TP-Link DDNS client in router settings
3. Set up a backup DDNS updater **on the NUC itself** (e.g., DuckDNS cron job) so WAN access doesn't depend on the router's DDNS client
4. Add a second SSH alias in `~/.ssh/config` for the backup DDNS hostname
5. User recalls having two DDNS systems, but only `tplinkdns.com` was found — investigate whether a second was ever set up
