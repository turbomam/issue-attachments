### bsdbng — Clean-room BugSigDB whiteroom

**Repo**: turbomam/bsdbng (updated Apr 13, all checks green)

A schema-first, guardrails-first reimplementation of the BugSigDB ingestion that the students are building organically in KG-Microbe-search.

**What it has**:
- LinkML schema with 5 nested classes: `BugSigDBDataset` > `StudyRecord` > `ExperimentRecord` > `SignatureRecord` > `TaxonRecord`
- Container class (`BugSigDBDataset`, `tree_root: true`) — validates a whole dataset in one file
- Enums for `direction` (increased/decreased) and `taxonomic_rank` (strain → superkingdom)
- Pattern validation: `taxon_id` must match `^NCBITaxon:[0-9]+$`, DOI must match `^10\..+$`
- Generated Pydantic models with field validators
- Full tooling: uv, just, ruff, mypy strict, pytest, bandit, deptry, pip-audit, linkml lint, CI, CodeQL
- Documented data access strategy: use raw CSVs from BugSigDB `Help:Export`, not pre-merged `full_dump.csv`

**What it doesn't have yet**:
- The CSV→YAML ingestion pipeline (only `assert_required_exports` stub exists)
- No raw data downloaded
- No per-study YAML outputs

### Key messages for students

**Your work proved the concept** — bsdbng gives it a permanent address. Infrastructure is already done. The issues that remain are scientific: taxon resolution, trait aggregation, phenotype crossref.

**Why guardrails matter when using LLMs**: The KG-Microbe-search repo had no automated checks — no linter, no type checker, no schema validation in CI. When LLM-generated code went into the repo, there was nothing to catch silent issues (wrong types, hardcoded paths, data quality bugs like defaulting direction to "increased"). Problems compounded across branches and students.

bsdbng fixes this by making `just check` the gate. It runs ruff, mypy strict, pytest, linkml lint/validate, bandit, deptry, and pip-audit. It doesn't care who wrote the code — human or LLM. If it passes, it's acceptable. If it doesn't, fix it before committing.

This isn't about not using LLMs. It's about integrating them into the dev/test ecosystem instead of copy-pasting from a browser. The loop should be: LLM generates code → `just check` → fix failures → commit. Whether you use Claude Code, Copilot, ChatGPT, or write it by hand, the same checks run.

**What went wrong without guardrails** (concrete examples from KG-Microbe-search):
- A ChatGPT session ruined a git branch (Anthea, Mar 4)
- LLM analysis outputs were validated after the fact, not during generation (Jed)
- Multiple students generating code that couldn't merge because no shared schema caught conflicts early
- Silent data quality bugs: direction defaulting to "increased" when unparseable, sentinel taxon IDs like NCBITaxon:0
- Six different data access patterns for the same BugSigDB source data

## bsdbng — Additional Fixes (2026-04-17)

### Taxon parsing was fundamentally wrong
BugSigDB encodes taxa as MetaPhlAn lineage paths:
- **Pipes** (`|`) separate lineage levels within one taxon (kingdom→...→leaf)
- **Semicolons** (`;`) separate distinct taxa within one signature

We were treating each pipe-separated ID as a separate taxon, inflating counts ~7x. Fixed to take the leaf (most specific) taxon only. MetaPhlAn rank prefixes (`k__`, `g__`, `s__`, etc.) now parsed for taxonomic rank.

### Comprehensive ingest logging
Every data drop and transformation assumption is now logged to `data/derived/ingest_log.json`. Summary: 1,923 studies written, 1,081 skips, ~199K info entries. All legitimate edge cases in BugSigDB.
