### Diagnostics

| Test | Result |
|------|--------|
| `nslookup mamillerpa.tplinkdns.com` | Resolves to 71.224.216.71 |
| `ping 71.224.216.71` | Responds, ~21ms RTT |
| `nc -zv -w 5 mamillerpa.tplinkdns.com 22` | Timed out |

### Diagnosis

The IP `71.224.216.71` pings but has zero open ports — it's almost certainly **no longer your home's public IP**. The ISP rotated the IP and the TP-Link router's built-in DDNS client didn't update `mamillerpa.tplinkdns.com`. The hostname now points to someone else's device.

This was working ~1 week ago from an urgent care office. The NUC itself is confirmed running (used locally within the hour).

### bsdbng — Clean-room BugSigDB whiteroom

**Repo**: turbomam/bsdbng (updated Apr 13, all checks green)

A schema-first, guardrails-first reimplementation of the BugSigDB ingestion that the students are building organically in KG-Microbe-search.

**What it has**:
- LinkML schema with 5 nested classes: `BugSigDBDataset` > `StudyRecord` > `ExperimentRecord` > `SignatureRecord` > `TaxonRecord`
- Container class (`BugSigDBDataset`, `tree_root: true`) — validates a whole dataset in one file
- Enums for `direction` (increased/decreased) and `taxonomic_rank` (strain → superkingdom)
- Pattern validation: `taxon_id` must match `^NCBITaxon:[0-9]+$`, DOI must match `^10\..+$`
- Generated Pydantic models with field validators
- Full tooling: uv, just, ruff, mypy strict, pytest, bandit, deptry, pip-audit, linkml lint, CI, CodeQL
- Documented data access strategy: use raw CSVs from BugSigDB `Help:Export`, not pre-merged `full_dump.csv`

**What it doesn't have yet**:
- The CSV→YAML ingestion pipeline (only `assert_required_exports` stub exists)
- No raw data downloaded
- No per-study YAML outputs

### bsdbng vs. student work — comparison

| Aspect | bsdbng | Students (KG-Microbe-search) |
|---|---|---|
| Schema design | Container class, inlined hierarchy, enums, patterns | Flatter schema, no container class (Jed's) |
| Data source | Raw CSVs from Help:Export (documented choice) | Mix of LLM outputs, pre-merged data, various formats |
| Tooling | uv, just, ruff, mypy strict, pre-commit, CI | Ad hoc — no shared build surface |
| Code quality | Strict typed Python package | Scripts in branches, organic growth |
| Artifacts | Canonical per-study YAML (planned) | Per-study folders with .txt/.yaml, TSVs, Google Slides |
| Validation | LinkML + Pydantic from day 1 | Jed using `linkml validate` after the fact |
| Progress | Schema done, pipeline not started | 1,743 studies analyzed, trait TSVs generated |

### The gap

bsdbng has what Mark's Feb 20 roadmap to Jed called "step 4" — container class, inter-class references, shared pattern. The students got to working outputs faster but with organic tooling. Bridging the two means:
1. Building the CSV→YAML pipeline in bsdbng
2. Reconciling Jed's schema with bsdbng's stricter schema
3. Validating the students' 1,743 study YAML outputs against bsdbng's schema
4. Eventually replacing ad hoc scripts with bsdbng as the canonical ingestion path

## bsdbng Pipeline — Assignment Plan

### Mark built: #27 + #28 + #34 (PR #35)

Pipeline is working end-to-end. PR: https://github.com/turbomam/bsdbng/pull/35

What was built:
- Download module with httpx, exponential backoff, Retry-After support, provenance sidecar
- Ingest pipeline: full_dump.csv → 1,925 per-study YAML files in ~60s
- NamedThing base class with unified `id` slot and per-class `id_prefixes`/patterns
- `pmid`, `doi`, `url` slots on StudyRecord
- Studies with no valid data are skipped and logged to stderr (no sentinels)
- Timed Justfile recipes logging to `logs/pipeline.log`
- Cleanup recipes for all steps
- Zero nosec/noqa/type:ignore suppressions
- All checks green (ruff, mypy strict, pytest, linkml lint/validate, bandit, deptry, pip-audit)

Schema design decisions:
- Bare StudyRecord per file (no container wrapper)
- `bsdb:` prefix (provisional — needs coordination with BugSigDB upstream, issue #38)
- Filenames use BSDB ID only; query YAML for PMID with yq
